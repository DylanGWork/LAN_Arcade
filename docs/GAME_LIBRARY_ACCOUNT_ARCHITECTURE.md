# LAN Arcade Game Library Account And Email Architecture

Last updated: 2026-06-28
Status: foundation in progress; full save migration is not implemented yet.


## 2026-06-28 Implementation Update

Confirmed live account primitives now exist in the Arcade API:

- account creation, sessions, local email metadata, favourites, recent activity, and save vault;
- friends and local messages/invites;
- `/arcade-api/server-info` capabilities include `account-friends` and `account-messages`.

These are not yet a complete Steam-style social layer. Treat them as the account foundation for later presence, game invites, parental controls, and save synchronization. Browser/emulator saves still need per-launcher proof before claiming family-safe isolation.

## Compatibility Assessment

Partially fits.

The deployed local mail server is a good foundation for LAN-only account email,
password reset, account notifications, and service-to-service SMTP. It already
runs Mailu with authenticated local submission, IMAPS, webmail, local aliases,
and an external-delivery block. The main blockers before account integration are
LAN reachability, account/mailbox provisioning automation, certificate trust,
backup/recovery completeness, and a final decision on child/family account rules.

## Current Arcade Account And Save Risk

The current game library is mostly static pages served from `/var/www/html/mirrors`
and generated by `/home/dylan/LAN_Arcade`. The private ROM and DOS vault launchers
are static EmulatorJS pages, for example:

- `/var/www/html/mirrors/private-rom-vault/play.html`
- `/var/www/html/mirrors/private-dos-vault/play.html`
- `/var/www/html/mirrors/emulatorjs-runtime/4.2.3/`

Current risk: saves, save states, emulator settings, achievements, and progress
are browser/profile storage unless a specific game has its own server-side
account system. A shared household laptop/browser can therefore overwrite or mix
progress between family members. The ROM library can remain shared, but launchers
and save storage need to become account-aware.

## Recommended Full Account Architecture

Use proper local LAN Arcade accounts, not anonymous profiles.

Account table fields should include at minimum:

```text
id, username, display_name, local_email, password_hash, role, status,
parent_account_id, recovery_email, created_at, updated_at, last_login_at
```

Recommended roles:

```text
admin, adult, child, service
```

The ROM/game library stays shared and read-only. All mutable user data must be
stored outside Git and outside the public mirror root, for example:

```text
/srv/lan-arcade/user-data/accounts/<account-id>/
  emulator-saves/
  emulator-states/
  emulator-settings/
  achievements/
  screenshots/
  per-game/<game-id>/
```

If `/srv` permissions are not ready, use a VM-local data path with the same shape
and document the migration target. Do not store account saves under
`/var/www/html/mirrors`, the repo, or the NFS native-download cache.

## Account To Game Mapping

Emulator and browser launchers:

- Require login before launching account-aware games.
- Prefix or scope EmulatorJS game IDs/save keys with the LAN Arcade account ID.
- Prefer server-side save sync/export where practical.
- Treat EmulatorJS save-key behavior as a limitation until tested per core.
- If a core cannot be isolated by launcher config, label it as requiring separate
  browser profiles or a patched runtime before it is considered family-safe.

Native/server games:

- Keep installers and server binaries shared.
- Map each game-specific account to `arcade_account_id`.
- Store game-specific credentials or generated service IDs server-side, not in
  static pages.
- For games with their own account systems, use the LAN Arcade email address as
  the email field where the game supports it.
- For single-player native games with local filesystem saves, document the save
  path and isolate by OS user, wrapper script, or per-account launch profile.

Achievements/settings:

- Store arcade achievements and launcher settings in the LAN Arcade account DB,
  keyed by account ID and game ID.
- Do not rely on browser localStorage as the source of truth once accounts exist.

## Email Usage

Local email address format:

```text
<username>@gannan.home.arpa
```

Rules:

- Normalize usernames to lowercase ASCII: `a-z`, `0-9`, `.`, `_`, `-`.
- Reserve aliases such as `admin`, `support`, `arcade`, `no-reply`, `postmaster`.
- Keep all account email under `gannan.home.arpa`.
- Do not allow external email addresses for password reset in the offline system.

Account creation:

- Create the LAN Arcade account first.
- Reserve or create the matching Mailu mailbox/alias second.
- Send a local welcome/verification message from `arcade@gannan.home.arpa` or
  `no-reply@gannan.home.arpa`.
- Verification proves access to the local mailbox only; it is not an external
  identity check.

Password reset:

- Generate one-use tokens, store only hashed tokens server-side, expire after
  15-30 minutes.
- Send reset links to local `gannan.home.arpa` addresses only.
- Links must point to the LAN arcade hostname, not a public URL.
- If the user loses arcade access and mailbox access, Dylan/admin performs manual
  recovery.

Admin recovery:

- Admins can force password reset, rotate account email, disable accounts, and
  reassign child recovery addresses.
- Record account recovery actions in an audit log.
- Mailu admin access is mailbox infrastructure recovery, not the normal arcade
  account recovery UI.

Notifications:

- Use local-only email for account changes, password resets, server start/stop
  notices, long-running game events, and admin warnings.
- Throttle notifications to avoid filling small mailboxes.
- Avoid noisy gameplay notifications by default.

Child/family accounts:

- Children should still have isolated LAN Arcade accounts and saves.
- Default child recovery should route to a parent/admin local email address.
- Decide per child whether they receive a real mailbox, an alias to a parent, or
  no direct mailbox with parent/admin recovery only.
- Add admin controls to disable password reset emails directly to young children
  if desired.

Service-to-service SMTP settings for the future account app:

```text
SMTP_HOST=127.0.0.1
SMTP_PORT=2587
SMTP_SECURITY=STARTTLS
SMTP_USER=arcade@gannan.home.arpa
SMTP_PASSWORD=<ARCADE_SMTP_PASSWORD_FROM_SECRET_STORE>
SMTP_FROM=arcade@gannan.home.arpa
SMTP_ALLOWED_RECIPIENT_DOMAIN=gannan.home.arpa
```

If the account app runs in a container, `127.0.0.1:2587` will point at that
container, not the host. Before implementation, choose one approved route:

1. Run the account service on the VM host and use loopback SMTP.
2. Attach the account service to an approved Mailu submission network/path.
3. Expose submission on the LAN IP only after Dylan explicitly approves LAN
   bindings, DNS, TLS, firewall, and trust handling.

Do not edit Mailu `data/state/admin` directly. Provision mailboxes/aliases via
Mailu admin UI, Mailu-supported CLI/API, or a future documented helper.

## Safeguards Against Save Loss

- Require account login before any launcher that can write saves or achievements.
- Keep the shared ROM/install library read-only to normal users.
- Store all mutable save paths under account IDs, not usernames alone.
- Use atomic writes or temp-file-then-rename for server-side save snapshots.
- Keep periodic save snapshots and retain several generations per account/game.
- Provide export/import for per-account saves before changing emulator runtime or
  launcher paths.
- Warn when a guest/shared-browser session is being used.
- For unverified emulator cores, mark save isolation as `needs-review` until a
  two-account overwrite regression test passes.


Minimum restore drill:

1. Restore mail state and secrets to a test path or test VM.
2. Restore arcade account DB and one account's save directory.
3. Log in as that account.
4. Open webmail or reset password locally.
5. Launch two emulator games and verify saves are still assigned to the correct
   account.

## Migration Path

Phase 1: keep existing static shelves public, but add optional LAN Arcade login
and account DB.

Phase 2: make EmulatorJS/DOS launchers account-aware and add per-account save
sync/export. Existing browser-local saves remain untouched until imported.

Phase 3: add a save import wizard: user logs in, opens the old local save source,
and imports into server-side account storage.

Phase 4: require login for save-capable launchers. Guest mode can remain for
stateless demos only.

Phase 5: map native/server games to LAN Arcade accounts and document exceptions
for games that cannot isolate progress.
